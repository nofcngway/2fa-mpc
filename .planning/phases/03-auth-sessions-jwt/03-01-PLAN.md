    1. Create `auth/internal/storage/redisstorage/session.go` implementing all 5 SessionStorage methods using the three-key Redis model per D-05:

       Key patterns:
       - `refresh_token:{jti}` -> JSON string `{"user_id":"...","token_family":"...","issued_at":"..."}` with TTL
       - `token_family:{family_uuid}` -> Redis Set of JTIs with TTL
       - `user_tokens:{user_id}` -> Redis Set of token_family UUIDs (no TTL per D-05)

       Methods:
       - `StoreRefreshToken(ctx, jti, userID, tokenFamily string, ttl time.Duration) error`:
         Use Redis Pipeline. SET `refresh_token:{jti}` with JSON-encoded RefreshTokenData and TTL. SADD `token_family:{family}` jti. EXPIRE `token_family:{family}` ttl. SADD `user_tokens:{userID}` family. Exec pipeline.

       - `GetRefreshToken(ctx, jti string) (*authService.RefreshTokenData, error)`:
         GET `refresh_token:{jti}`. If redis.Nil, return nil, nil. JSON-unmarshal into RefreshTokenData. Return data.

       - `DeleteRefreshToken(ctx, jti string) error`:
         First GET the token data to find its family. DEL `refresh_token:{jti}`. SREM `token_family:{family}` jti. Check SCARD `token_family:{family}` -- if 0, also SREM `user_tokens:{userID}` family and DEL `token_family:{family}`. Use pipeline for atomicity.

       - `DeleteTokenFamily(ctx, family string) error`:
         SMEMBERS `token_family:{family}` to get all JTIs. Pipeline DEL for each `refresh_token:{jti}`. DEL `token_family:{family}`. Note: caller handles removing from user_tokens. Make idempotent (no error if keys already deleted per Pitfall 2 from RESEARCH.md).

       - `DeleteAllUserTokens(ctx, userID string) error`:
         SMEMBERS `user_tokens:{userID}` to get all families. For each family: SMEMBERS to get JTIs, DEL each `refresh_token:{jti}`, DEL `token_family:{family}`. Then DEL `user_tokens:{userID}`. Use pipeline per D-11.

       Import `authService` package for `RefreshTokenData` type. Use `encoding/json` for serialization.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> Auth gRPC | Untrusted token strings cross here for parsing/validation |
| Auth -> Redis | Session data stored/retrieved; Redis assumed trusted within network |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Spoofing | jwt.go ParseToken | mitigate | `jwt.WithValidMethods([]string{"RS256"})` enforced in ParseToken -- prevents alg:none and alg:HS256 confusion attacks (SEC-01, D-04) |
| T-03-02 | Tampering | jwt.go GenerateAccessToken | mitigate | RSA-2048 private key used for signing; key loaded from file at startup, never logged (D-17) |
| T-03-03 | Information Disclosure | domain/errors.go | mitigate | Generic `ErrInvalidCredentials` for both wrong email and wrong password -- no credential enumeration |
| T-03-04 | Information Disclosure | redisstorage/session.go | accept | Redis data not encrypted at rest -- acceptable for academic project; Redis on localhost only |
| T-03-05 | Denial of Service | redisstorage/session.go | accept | `user_tokens:{user_id}` set has no TTL and grows unbounded -- acceptable for academic project scope per RESEARCH.md open question #2 |
</threat_model>
