Контекст задачи:
Нужно спроектировать и затем реализовать frontend для проекта.

Обязательные страницы:
- Login page
- Register page
- Home page
- 2FA Setup / Code Binding page

Функциональные требования:
- на странице 2FA должен использоваться компонент InputOTP для ввода OTP-кода
- также должен быть альтернативный сценарий: пользователь может ввести backup code вместо OTP
- нужно продумать UX так, чтобы пользователь мог явно выбрать или переключить способ подтверждения: OTP или backup code
- необходимо предусмотреть корректные состояния:
  - default
  - focus
  - filled
  - error
  - loading
  - success
  - disabled

Требования к визуальному стилю:
Нужно сделать UI в стиле Apple-inspired Liquid Glass.

Для Liquid Glass обязательно использовать и учитывать:
- translucent glass surfaces
- soft blur / background diffusion
- reflective highlights
- layered depth
- rounded corners
- subtle gradients
- thin separators
- elegant spacing

Дополнительные визуальные требования:
- premium, calm, futuristic, clean look
- стеклянные полупрозрачные панели
- мягкие световые блики
- аккуратные слои и ощущение глубины
- большие скругления
- минималистичная типографика
- деликатные градиенты
- мягкие тени и внутренние световые акценты
- чистая и дорогая визуальная подача
- современный, консистентный интерфейс
- высокий уровень читаемости
- не делать generic dark glassmorphism
- не делать cyberpunk / neon aesthetics
- не делать перегруженный интерфейс

Темы:
- обязательно реализовать light theme и dark theme
- переключение тем должно быть реализовано через HeroUI
- обе темы должны сохранять единый visual language Liquid Glass
- нужно продумать токены и поведение тем так, чтобы стеклянные эффекты выглядели уместно и в светлой, и в тёмной теме

Архитектурные требования:
1. Архитектура должна быть чистой, масштабируемой и пригодной для дальнейшего роста проекта.
2. Нужно выстроить понятную иерархию:
   component -> widget -> page
3. Обязательно использовать alias imports вместо длинных относительных путей.
4. Все компоненты должны проектироваться не минимально “под текущий экран”, а как полноценные переиспользуемые сущности.
5. Каждый компонент должен быть продуман заранее:
   - с чистым API
   - с максимально полным и расширяемым набором props
   - с поддержкой variants
   - с поддержкой sizes
   - с поддержкой states
   - с поддержкой disabled/loading/error/success states, где это уместно
   - с учетом дальнейшего переиспользования
6. Компоненты не должны быть одноразовыми без причины.
7. Нужно построить основу design system:
   - colors
   - typography
   - radius
   - blur levels
   - shadows
   - spacing
   - borders
   - component variants
8. Нужно соблюдать разделение ответственности между слоями:
   - components — базовые переиспользуемые UI-примитивы
   - widgets — композиции компонентов под конкретные блоки интерфейса
   - pages — страницы, собирающие widgets и page-level логику
